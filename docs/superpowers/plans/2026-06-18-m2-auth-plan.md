# Plano de Implementação — M2: Autenticação (JWT + RBAC)

**Spec:** [2026-06-18-helpdesk-pro-design.md](../specs/2026-06-18-helpdesk-pro-design.md)
**Detalhes técnicos:** [`prompt-helpdesk-claude-code.md`](../../../prompt-helpdesk-claude-code.md) (Fases 3.1, 3.2, 3.6, 3.7 e 4.5)
**Objetivo do marco:** logar com os 3 usuários demo, com access token (15min) +
refresh token (7d) persistido no banco, RBAC por papel e refresh automático no
frontend ao receber 401.

## Definição de "pronto" (acceptance)

1. `POST /api/auth/register` cria usuário (senha com hash) e retorna tokens.
2. `POST /api/auth/login` valida credenciais e retorna access + refresh.
3. `GET /api/auth/me` retorna o usuário autenticado (sem senha); 401 sem token.
4. `POST /api/auth/refresh` troca um refresh válido por um novo par.
5. `POST /api/auth/logout` invalida o refresh token no banco.
6. `rbac.middleware` bloqueia papéis não autorizados com 403.
7. Frontend: tela de login funcional, store Zustand persistido, interceptor
   Axios que tenta refresh automático no 401 e desloga se falhar.

> Verificação end-to-end depende do banco (M1 adiado). O código será validado
> por typecheck/build; o teste manual roda quando houver Postgres.

## Backend

### Camada de infra/config
- `src/config/env.ts` — carrega e valida variáveis de ambiente.
- `src/config/database.ts` — instância única do PrismaClient.
- `src/utils/ApiError.ts` — erro custom com statusCode.
- `src/utils/jwt.ts` — `signAccess`, `signRefresh`, `verifyAccess`, `verifyRefresh`.
- `src/utils/bcrypt.ts` — `hash`, `compare` (bcryptjs).

### Módulo auth (`src/modules/auth/`)
- `auth.schema.ts` — Zod: registerSchema, loginSchema, refreshSchema.
- `auth.service.ts` — register, login, refresh (rotação), logout, me.
- `auth.controller.ts` — adapta HTTP ↔ service, sem regra de negócio.
- `auth.routes.ts` — `/register /login /refresh /logout /me`.

### Middlewares (`src/middlewares/`)
- `auth.middleware.ts` — lê Bearer, valida JWT, popula `req.user`.
- `rbac.middleware.ts` — `requireRole(...roles)` → 403.
- `error.middleware.ts` — formata ApiError/Zod/Prisma/JWT.
- `validate.middleware.ts` — aplica um schema Zod ao body.

### App/server
- `src/app.ts` — Express com helmet, cors, json, rotas, error handler.
- `src/server.ts` — sobe o app na `PORT`.
- Deps novas: `express`, `cors`, `helmet`, `jsonwebtoken`, `zod` (+ `@types/*`).

## Frontend (scaffold mínimo do Vite chega aqui)

- Scaffold: `package.json`, `vite.config.ts`, `tailwind.config.ts`,

## Riscos / atenção
- Sem banco, não há teste de integração agora; manter o código compilando.
- Refresh token rotation: ao usar um refresh, invalidar o antigo e emitir novo
  (evita replay). Guardar `expiresAt` no banco e checar.
- Não logar segredos; `JWT_SECRET`/`JWT_REFRESH_SECRET` só via env.
